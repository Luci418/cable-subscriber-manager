# Release Notes — v0.9.0

_Released 2026-06-05_

## Theme

**Provider-first architecture.** Every service-bearing record (pack,
transaction, active subscription) now knows *which upstream source* it
belongs to. This unlocks per-provider revenue attribution, exposure
tracking, and future upstream-cost reconciliation.

See ADR-007 in [`../ARCHITECTURE_DECISIONS.md`](../ARCHITECTURE_DECISIONS.md)
for the design rationale.

## Features added

- New `providers` entity (per operator, scoped by Service Type).
- Provider Management dialog under `Manage → Providers`, reachable from the
  main subscriber screen.
- Provider Performance card on the Analytics page:
  - Active subscribers by provider
  - Revenue by provider (from `payment` transactions)
  - Outstanding balance by provider
- Provider selector in:
  - Pack management (required)
  - Add / Edit Transaction (shown only when the service has >1 active
    provider; defaults to the subscriber's current provider for that
    service)

## Features changed

- Pack creation now **requires** a provider. Existing packs were backfilled.
- Subscription assignment now stores `providerId` and `providerName` in the
  subscription blob and updates the subscriber's per-service provider column
  (`cable_provider_id` or `internet_provider_id`) on activation/renewal.

## Migrations performed

Migration: `supabase/migrations/20260605090507_*.sql`

- Created `public.providers` (with RLS + GRANTs to `authenticated`).
- Added `provider_id` to `packs`, `transactions`.
- Added `cable_provider_id`, `internet_provider_id` to `subscribers`.
- For each operator: created `Default Cable Network` and `Default Internet`
  provider rows; assigned existing packs, transactions, and active
  subscriptions to the appropriate default.
- Added `public.is_provider_in_use(provider_uuid uuid)` RPC to block
  destructive deletes.

The migration is **additive** and was designed to be safe to apply ahead of
the new frontend going live.

## Risks

- **Default provider rename.** Operators who haven't renamed the auto-created
  defaults to real provider names will see "Default Cable Network" in
  analytics. This is cosmetic only — the data is correct.
- **Provider mismatch on legacy transactions.** Transactions created before
  this release are tagged with the default provider for their service. If an
  operator wants accurate historical attribution, they can edit individual
  transactions; bulk re-attribution is not provided (and probably not worth
  the effort).
- **Renewal still looks like a new sale.** Provider attribution is correct,
  but the renewal-vs-new-sale distinction is still pending (Phase 6).

## Upgrade notes

1. Apply the migration before deploying the new frontend.
2. Sign in as the operator and visit `Manage → Providers` to rename the
   default providers to your real upstream names (e.g. "BSNL Broadband",
   "Fastnet", "Internal Cable Network").
3. (Optional) Edit recent transactions where you want explicit provider
   attribution instead of the default.
4. Verify the Provider Performance card on the Analytics page renders.

## Rollback

The migration is additive; rolling back the frontend alone is safe (legacy
UI ignores the new columns). Reverting the migration is **not** recommended
once any new subscription has been recorded with provider attribution — you
would lose that attribution.
