# Lifecycle Integrity Audit — June 2026

**Scope:** Operational integrity review of every business object in the system.
This document records **current implementation behavior** (not desired behavior)
based on the schema, SQL functions/triggers, hooks, and UI workflows as of
commit following the Tier-0 hardening migration
(`20260607065719_*_tier0_hardening.sql`).

**Output:** Risks first, recommendations second. No code was changed as part
of this audit. Section §13 lists the prioritized follow-up work.

> **Critical pre-existing bug surfaced by Tier-0 (see §3.5, §5.5, §13.A0).**
> The new `transactions_recalc_balance_trg` trigger recomputes
> `cable_balance` / `internet_balance` from the ledger on every transaction
> INSERT/UPDATE/DELETE. Several UI paths still perform a *manual* balance
> mutation on the subscriber row in addition to writing the transaction —
> this now **double-counts**. Listed as A0 (must-fix before production).

---

## Conventions used in this document

- **State** = a value/combination of values that defines where the object is
  in its lifecycle (e.g. `active`, `expired`).
- **Transition** = a state change driven by user action, RPC, trigger, or
  cron.
- **Hard delete** = `DELETE FROM ...` (row physically gone).
- **Soft delete / archive** = the row stays; a flag (`is_active=false`,
  `status='cancelled'`, etc.) excludes it from operational views.
- **Risk rating** = `Low / Medium / High / Critical`. Critical = data loss,
  financial discrepancy, or audit-failure risk in production use.

## 1. Subscriber

### 1.1 Overview
Represents a household/customer holding zero or more services (cable,
internet). Single wide row, stored balances per service, JSONB blobs for
current subscription, arrays of JSONB blobs for subscription history.

Relationships:
- `transactions.subscriber_id` → subscriber  (`ON DELETE CASCADE`)
- `complaints.subscriber_id` → subscriber    (`ON DELETE CASCADE`)
- `stb_inventory.subscriber_id` → subscriber (`ON DELETE SET NULL`)
- Embeds: `current_subscription`, `internet_subscription`,
  `subscription_history[]`, `internet_subscription_history[]`,
  `cable_balance`, `internet_balance`, `current_pack`,
  `current_internet_pack`, `cable_provider_id`, `internet_provider_id`,
  `stb_number`, `services[]`.

### 1.2 States
There is **no explicit subscriber state column**. Effective states are
derived:

| Derived state          | How it is recognized in data                       |
| ---------------------- | -------------------------------------------------- |
| New / no services      | `services = []` or both subscription blobs `NULL`  |
| Active (any service)   | At least one of `current_subscription` / `internet_subscription` not null and `endDate > now()` |
| Lapsed                 | Subscription blob present but `endDate <= now()` (transient — cron / load moves it to history) |
| Inactive               | All subscription blobs null, history not empty     |
| Deleted                | Row removed                                        |

There is **no `archived` or `inactive` flag** on the row.

### 1.3 Creation rules
- `name` non-blank, `mobile` digits 7–15, `subscriber_id` auto-generated by
  `generate_subscriber_id(region)` RPC (advisory-locked).
- Mobile uniqueness is **enforced per operator** (`uq_subscribers_user_mobile`).
- A region must already exist (free-text field, but UI is dropdown).
- STB serial — if provided — must already exist in `stb_inventory` (trigger
  `sync_stb_inventory_on_subscriber_change` claims it).

### 1.4 Edit rules
All fields are freely editable through `EditSubscriberDialog`, including
`subscriber_id`, region, mobile, services list, STB number, and the balance
columns are editable as side effects of other actions.

Safeguards in place:
- Mobile format CHECK constraint.
- STB swap is handled by the inventory trigger (releases old, claims new).
- Balance columns are stored & overwritten by the new ledger trigger (§5).

Gaps:
- `subscriber_id` (the human-readable code) is editable — bypasses
  `generate_subscriber_id` uniqueness logic for the new value. Risk:
  Medium. Recommendation: make read-only in UI after creation, or run it
  through a renaming RPC.
- No audit columns on the subscriber row (`created_by`, `edited_at`).

### 1.5 Deletion rules (CRITICAL — current behavior)
`SubscriberDetail` → `Delete` button → `AlertDialog` → `useSubscribers.deleteSubscriber()`
→ raw `DELETE FROM subscribers WHERE id=?`.

**There are no application-level safeguards.** A subscriber can currently be
hard-deleted with:

| Pre-existing condition            | Allowed today? | Effect                                                |
| --------------------------------- | -------------- | ----------------------------------------------------- |
| Active cable subscription         | ✅ Yes          | Subscription blob lost                                |
| Active internet subscription      | ✅ Yes          | Subscription blob lost                                |
| Outstanding balance (debt)        | ✅ Yes          | Debt vanishes, no audit trail                         |
| Credit balance (operator owes)    | ✅ Yes          | Credit vanishes, customer has no record               |
| Assigned STB                      | ✅ Yes          | STB row's `subscriber_id` → NULL but **status stays `assigned`** (orphan) |
| Assigned ONU/Router               | ✅ Yes          | Same orphan as above                                  |
| Open complaints                   | ✅ Yes          | All complaints CASCADE-deleted                        |
| Closed complaints (history)       | ✅ Yes          | All complaints CASCADE-deleted                        |
| Transaction history (any)         | ✅ Yes          | All transactions CASCADE-deleted → **revenue history is rewritten retroactively** |

**Risk: Critical.** A single mis-click silently rewrites revenue, churn,
collections, and provider attribution analytics, and orphans hardware
inventory.

### 1.6 Archive rules
Archive functionality **does not exist** today. Should it? Yes — a soft
`status` (`active` / `suspended` / `closed`) is required to:
- Stop showing inactive accounts in collection lists without losing history.
- Preserve transaction & subscription history for analytics.
- Allow a "reopen" workflow without re-creating IDs.

### 1.7 Dependencies on delete
| Dependent table | FK action       | Result                                |
| --------------- | --------------- | ------------------------------------- |
| transactions    | CASCADE         | All deleted; trigger fires per row but balance target is gone — silent loss |
| complaints      | CASCADE         | All deleted                           |
| stb_inventory   | SET NULL        | `subscriber_id` cleared, status NOT reset |

### 1.8 Analytics impact on delete
Every revenue, ARPU, churn, provider-attribution, collections-by-route,
and region metric **changes retroactively**. There is no anonymized
"deleted subscriber" placeholder. Severity: **Critical** for any operator
running monthly/quarterly reports.

### 1.9 Invalid states currently possible
- STB row in `stb_inventory` with `status='assigned'` and `subscriber_id IS NULL`
  (post-subscriber-delete orphan).
- Subscriber edited to a `subscriber_id` that violates the generator format.
- A subscriber with `current_subscription` set, `endDate` in the past, before
  the cron/load expiry runs (intentional but transient — acceptable).

### 1.10 Risk rating: **Critical** (deletion path).

## 2. Subscription (embedded JSONB)

### 2.1 Overview
Subscriptions are not their own table. They are JSONB documents inside
`subscribers.current_subscription` /
`subscribers.internet_subscription` / their `*_history` arrays. ADR-004
captures the rationale.

### 2.2 States
| Status                | Where stored                              |
| --------------------- | ----------------------------------------- |
| `active`              | `current_subscription` / `internet_subscription` |
| `expired`             | `*_history[]` (moved by `expire_lapsed_subscriptions`) |
| `cancelled`           | `*_history[]` (moved by cancel-with-refund) |
| (no `suspended` state today) |                                    |

### 2.3 Creation rules
- Cannot create a new one while an active subscription exists for the same
  service (`AddPackageSubscriptionDialog` blocks).
- Auto-creates a `charge` transaction for `packPrice × duration`.

### 2.4 Edit rules
- **No edit UI exists.** A subscription blob is immutable from creation to
  termination. Field changes happen only through cancel-and-reassign.

### 2.5 Delete rules
- A subscription cannot be "deleted" individually through the UI.
- An admin editing the JSONB blob directly (DB console) would bypass *all*
  ledger reconciliation. Not enforced by RLS or trigger.

### 2.6 Dependencies & analytics
- Each subscription emits a `charge` transaction at assignment (and a
  `refund` on cancel). The charge is **not** automatically reversed if the
  subscription blob is mutated/deleted out-of-band.
- Renewals are currently indistinguishable from new sales (`BUSINESS_RULES.md`
  §4.5). Churn analytics is approximate.

### 2.7 Invalid states currently possible
- Active subscription with `endDate` in the past for >1 hour (cron failure).
- History entry without a matching charge transaction (after a
  trigger-bypassing edit).
- A duplicate active entry in the history array with the same `id` — code
  attempts to dedupe but trusts the `id` field.

### 2.8 Risk rating: **Medium**. The model is locked enough that day-to-day
flows are safe, but there is no defensive check against direct JSONB
tampering.

## 3. Transaction

### 3.1 Overview
Append-only ledger entry. Three types: `charge`, `payment`, `refund`.
Tagged with `subscriber_id`, `service_type`, `provider_id`.

### 3.2 States
None. A transaction is either present or deleted. There is **no `voided`
state** today.

### 3.3 Creation rules
- `amount >= 0` (CHECK).
- `service_type ∈ {cable, internet}` (CHECK).
- `created_by` and `created_at` stamped by `transactions_audit_stamp` trigger.
- Balance recomputed by `transactions_recalc_balance_trg`.

### 3.4 Edit rules
- UI exposes `EditTransactionDialog`: amount, type, service, description,
  provider, date are all editable.
- **No restriction on changing `type` or `service_type`** post-creation —
  contradicts `BUSINESS_RULES.md` §5 ("should not change after creation").
- `edited_at` / `edited_by` stamped by trigger.

### 3.5 Deletion rules — **DOUBLE-COUNT BUG (A0)**
`SubscriberDetail.handleDeleteTransaction`:

```ts
await supabase.from('transactions').delete().eq('id', transaction.id);
// then:
await applyBalanceChange(svc, balanceDelta(transaction.type, amount, -1));
```

The DB-level trigger `transactions_recalc_balance_trg` already recomputes
the balance from the remaining ledger. The client then **applies a manual
delta on top**, so the stored balance drifts by exactly the deleted
transaction's amount on every delete.

The same pattern exists for:
- `EditTransactionDialog` save path (`handleUpdateTransaction`,
  lines 209–215 in `SubscriberDetail.tsx`).
- Cancel-with-refund (`handleCancelSubscription`, lines 256–286): manually
  decrements `cable_balance` *and* inserts a refund transaction whose
  trigger then recomputes the same field.

**Risk: Critical.** Every transaction edit, transaction delete, and
subscription cancellation since the Tier-0 migration produces silent
balance drift in opposite directions.

### 3.6 Archive rules
None. No void / reversal workflow. The recommended pattern for an audited
ledger is to *never* delete, only post an offsetting transaction. Not
implemented.

### 3.7 Dependencies & analytics
- Drives `cable_balance` and `internet_balance` via trigger.
- Drives all revenue and collections analytics in `pages/Analytics.tsx`.
- Drives provider revenue attribution (Phase 1).

### 3.8 Invalid states currently possible
- A `charge` transaction with no matching subscription history entry
  (manual-charge UI path — intentional but unlabelled).
- A `payment` whose `service_type` was edited from `cable` → `internet`
  after creation, leaving the cable history visually unbalanced.
- Transactions whose `provider_id` references a now-deleted provider — not
  possible via UI (blocked) but possible via direct SQL.

### 3.9 Risk rating: **Critical** (because of §3.5).

## 4. Provider

### 4.1 Overview
Tagging entity for service source (BSNL, Fastnet, etc.). Phase 1.

### 4.2 States
`is_active = true | false`. No other states.

### 4.3 Creation / edit / delete
- Create: name + service_type, unique per (user, service_type, name).
- Edit: name, notes, service_type, is_active. **`service_type` is editable
  post-creation** — risky, because changing it can desync packs /
  transactions tagged with this provider. Risk: Medium.
- Delete: blocked by `is_provider_in_use` RPC if any pack, transaction, or
  subscriber row references it. ✅ Safe.
- Retire (`is_active=false`): supported. ✅ Correct.

### 4.4 Archive rules
"Retire" = soft delete via `is_active=false`. Historical references remain
valid. This is the right model.

### 4.5 Invalid states currently possible
- A provider whose `service_type` was flipped after packs were already
  tagged to it. The packs keep the original provider_id but the provider
  now claims a different service. Recommend: make `service_type` read-only
  after first use.

### 4.6 Risk rating: **Low-Medium**. Delete is safe; edit of
`service_type` is the only loose thread.

## 5. Pack / Plan

### 5.1 Overview
Catalog item: name, price, channels, validity, billing_type, service_type,
provider_id. Snapshotted into subscription blobs at sale time.

### 5.2 States
`is_active = true | false`. UI exposes "Retire" and "Reactivate".

### 5.3 Creation
- Unique by `(user_id, service_type, name)` — but DB also still has a stray
  unique constraint on `(name)` only (`packs_name_key`) which **breaks
  multi-operator usage** if two operators ever share the DB. Risk: Medium
  *today only because the DB is single-tenant*; **High** if multi-tenant
  is ever enabled.
- Prepaid packs require `validity_days > 0` (CHECK).
- `price >= 0` (CHECK).

### 5.4 Edit
All fields editable, including `price` and `service_type`. Editing `price`
does **not** retroactively change historical charges (good — snapshot in
blob). Editing `service_type` of a pack in use is allowed and will desync
displayed history (the blob's `service_type` was captured at sale, but
joins/lookups by pack name will now mismatch). Risk: Low (cosmetic).

### 5.5 Deletion
- Blocked by `is_pack_in_use` if any current subscription references the
  pack name. ✅ Correct for active subscriptions.
- **Not blocked** if only *historical* subscriptions or transactions
  reference the pack name. Hard-deleting then leaves transaction
  descriptions pointing at a missing pack and the Pack Analytics tab
  showing "Unknown pack". Risk: Medium.
- Recommendation: switch hard-delete to soft-retire whenever any
  transaction or subscription history mentions the pack.

### 5.6 Archive rules
"Retire" hides from dropdowns, keeps the row. Right model.

### 5.7 Risk rating: **Medium** (stray unique constraint + history-aware
delete check missing).

## 6. STB / ONU / Router (`stb_inventory`)

### 6.1 Overview
Physical device with serial number, device_type (`stb`/`onu`/`router`),
service_type, status, optional `subscriber_id`.

### 6.2 States
`available`, `assigned`, `faulty`, `decommissioned`. Backed by enum-ish
column (Postgres `stb_status`).

### 6.3 Transitions
| From → To           | Trigger                                 |
| ------------------- | --------------------------------------- |
| available → assigned | `assignStb` or subscriber STB-field trigger |
| assigned → available | `unassignStb` or subscriber edit/delete (FK SET NULL) |
| any → faulty         | `markAsFaulty` (auto-unassigns if assigned) |
| faulty → available   | `markAsRepaired`                       |
| available/faulty → decommissioned | `decommission`                |

### 6.4 Creation
- Serial must be unique per operator.
- `device_type` must match `service_type` (CHECK).

### 6.5 Edit
All fields editable, including `serial_number` and `service_type`.
Editing `service_type` of an assigned device that still references the
original service is unguarded.

### 6.6 Deletion — **partial safeguard, but orphan path exists**
- `deleteStb` blocks deletion when status is `assigned`. ✅
- BUT: the subscriber-delete path (§1.5) uses `ON DELETE SET NULL` on the
  FK. The device's `subscriber_id` becomes NULL while `status` stays
  `assigned` → **orphan state**.
- A device can be hard-deleted from `available` / `faulty` /
  `decommissioned`, erasing the historical assignment trail entirely.
  Risk: Medium for audit & warranty tracking.

### 6.7 Invalid states currently possible
- `status='assigned'` with `subscriber_id IS NULL` (subscriber-delete orphan).
- `status='available'` with `subscriber_id != NULL` (possible if
  `updateStb` is called with one but not the other — UI prevents but DB
  doesn't enforce).

### 6.8 Risk rating: **Medium**.

## 7. Complaint

### 7.1 Overview
Issue logged against a subscriber. Status workflow `pending` →
`in-progress` → `resolved`.

### 7.2 States
`pending`, `in-progress`, `resolved` (CHECK).
`resolved` requires `resolved_date IS NOT NULL` (CHECK).

### 7.3 Creation / edit
- Description non-blank (CHECK).
- Category, priority constrained.
- Updates allow free transitions between any of the three states.

### 7.4 Deletion
- Hard delete via `deleteComplaint`, no guards.
- CASCADE on subscriber delete — complaint history vanishes silently.

### 7.5 Risk rating: **Low** (limited financial impact), Medium for
auditing if subscriber deletion is unrestricted.

## 8. Region

### 8.1 Overview
Free-text grouping label used to bucket subscribers and feed the
`generate_subscriber_id` prefix.

### 8.2 States
None. No active/retired flag.

### 8.3 Creation / edit
- Unique per `(user_id, name)`. ✅
- No edit UI exists — only add + delete.

### 8.4 Deletion
- Blocked by `is_region_in_use` if any subscriber references the region by
  name. ✅
- A region rename would orphan the subscriber→region link (no
  referential integrity — it's by name, not FK). Today there is no rename
  UI, so risk is theoretical.

### 8.5 Risk rating: **Low**.

## 9. User (`auth.users` + `profiles`)

### 9.1 Overview
One operator account. `profiles` row is created via `handle_new_user`
trigger. RLS scopes every other table by `user_id`.

### 9.2 States
- `auth.users` row: active. No "disabled" or "suspended" workflow.
- `profiles.id` FK has `ON DELETE CASCADE` → deleting the auth user
  cascades through every domain table.

### 9.3 Deletion
**Hard-deleting an operator wipes their entire dataset.** No undo, no
export, no archive. The Lovable Cloud console allows this via the auth UI.
Risk: **High** for production — needs a documented backup-before-delete
runbook.

### 9.4 Risk rating: **High** (operational, not UI-driven).

## 11. Invoice / Receipt

### 11.1 Overview
There is **no `invoices` table**. Receipts are PDFs generated on demand
from `transactions` + subscriber data via `src/lib/pdf.ts`. ADRs note
this is intentional for current scale.

### 11.2 Lifecycle
- A "receipt" exists only as a PDF in the user's browser at the moment of
  generation. Re-printing re-derives from the underlying transaction.
- If the source transaction is deleted/edited, the regenerated receipt
  will silently differ from any printed copy. No serial number, no audit
  link.
- Risk: Medium-High once operators start handing physical receipts to
  customers. Tracked in `FUTURE_EVOLUTION.md` (Invoice entity).

## 12. Cross-cutting findings

### 12.1 Stored balance trustworthiness
The Tier-0 trigger guarantees `cable_balance` / `internet_balance` =
`Σ(charges − payments + refunds)` per service **as long as nothing else
writes the balance column**. Multiple UI paths still do (see §3.5). The
result is **negative trust**: the stored balance is now provably wrong
after most edits.

### 12.2 Subscriber-delete cascade fan-out
A single `DELETE FROM subscribers` cascades through transactions and
complaints, sets STB FKs to NULL, and produces silent analytics rewrites
plus inventory orphans. There is no application-layer pre-check.

### 12.3 No object has a uniform `archived` flag
- `packs.is_active`, `providers.is_active`, `stb_inventory.status` work.
- `subscribers`, `subscriptions`, `transactions`, `complaints`, `regions`
  have no soft-state equivalent.

### 12.4 Audit columns coverage
Only `transactions` has `created_by` / `edited_at` / `edited_by` (added in
Tier-0). Subscribers, subscriptions, complaints, packs, providers, STBs
have none. For a single-operator deployment this is acceptable; for
shared/staff deployments it is not.

### 12.5 RLS scope correctness
All public tables enforce `auth.uid() = user_id`. The `complaints` join
in `useComplaints` relies on RLS to scope the embedded `subscribers`
record — verified safe today.

## 13. Prioritized remediation roadmap

> "A0" = release-blocker, must be fixed before any production-grade use.

### Tier A0 — Data-integrity must-fix

| ID  | Item | Effort | Risk if skipped |
| --- | ---- | ------ | --------------- |
| **A0.1** | Remove all client-side balance mutation in `SubscriberDetail` (`handleDeleteTransaction`, `handleUpdateTransaction`, `handleCancelSubscription`). Let the DB trigger be the single source of truth. | S (1–2 hrs) | Critical — silent balance drift on every edit/cancel |
| **A0.2** | Block `subscribers` hard-delete when *any* of: active subscription, non-zero balance, assigned device, any transaction row exists. Add a `deactivate` action that nullifies subscriptions and marks `status='closed'`. | M (½ day) | Critical — silent revenue rewrite |
| **A0.3** | When a subscriber is closed/deleted, explicitly mark assigned devices `available` (or `returned`) and clear `subscriber_id` in one transaction — fixes the orphan-device state in §6.7. | S | Medium |
| **A0.4** | Drop stray `packs_name_key` unique constraint (replaced by `uq_packs_user_name_service`). | S | High in any multi-tenant scenario |

### Tier A1 — Within 30 days

| ID  | Item | Effort |
| --- | ---- | ------ |
| A1.1 | Add `subscribers.status` enum (`active`/`suspended`/`closed`) + archive UI; exclude closed from collection lists; keep them in analytics. | M |
| A1.2 | Make `subscribers.subscriber_id` read-only after creation; route renames through an RPC that re-checks uniqueness. | S |
| A1.3 | Make `providers.service_type` and `packs.service_type` read-only after any reference exists. | S |
| A1.4 | Replace transaction hard-delete with a "void" workflow that posts an offsetting transaction and flags the original as `voided` (new boolean / status). Hard-delete remains DB-only. | M |
| A1.5 | Extend `is_pack_in_use` to also count `transactions` and historical subscription entries; only allow hard-delete when zero history exists. | S |
| A1.6 | Add `complaint` deletion guard: block if `status='resolved'` (preserve audit), allow archive of `pending`/`in-progress` only. | S |

### Tier A2 — When the dataset / team grows

- Add `created_by` / `updated_by` to `subscribers`, `packs`, `providers`,
  `complaints`, `stb_inventory`.
- Introduce a real `invoices` table once any regulator or customer asks
  for a verifiable receipt.
- Reconcile-balances RPC + daily report (already on the Tier-1 list in
  `REVIEW_RESPONSE_2026-06.md`).
